# ADR 0001: Границы микросервисов, BFF, сессии и межсервисная идентичность

## Статус

Принято

## Контекст

Продукт — веб-приложение планирования питания с пятью ограниченными контекстами (см. [domain-contexts-event-storming.md](../domain-contexts-event-storming.md)). Требуется:

- отдельные деплой-единицы по доменным границам;
- единая точка входа для браузера;
- аутентификация email/пароль с данными на сервере ([business-doc.md](../business-doc.md) §4);
- изоляция данных пользователя.

## Решение

### 1. Набор микросервисов (1:1 с bounded context)

| Сервис | Контекст |
|--------|----------|
| `identity-access` | Identity & Access |
| `recipe-catalog` | Recipe Catalog |
| `recipe-import` | Recipe Import |
| `meal-planning` | Meal Planning |
| `shopping-list` | Shopping List |

`recipe-import` вынесен отдельно от `recipe-catalog`, чтобы изолировать исходящий HTTP, таймауты и сбои парсинга от CRUD каталога.

### 2. API Gateway

Публично доступен один хост (TLS). Gateway маршрутизирует префиксы (например `/api/iam`, `/api/catalog`, `/api/import`, `/api/planning`, `/api/shopping`) к соответствующим сервисам, задаёт таймауты (в частности для import) и при необходимости rate limiting.

### 3. BFF (Backend for Frontend)

Отдельный компонент для веб-клиента (Vue микрофронты):

- принимает запросы браузера с cookie сессии;
- агрегирует ответы нескольких микросервисов под сценарии UI ([design-plan.md](../design-plan.md));
- не раскрывает браузеру внутренние адреса микросервисов;
- пробрасывает **correlation id** (`X-Correlation-Id`) во внутренние вызовы.

Браузер обращается к API преимущественно через `/bff/v1/...` (точные пути — [bff-routes.md](../../contracts/bff-routes.md)).

### 4. Сессии: httpOnly cookie + Redis

- После успешного входа `identity-access` создаёт запись сессии в **Redis** (ключ сессии → `user_id`, срок жизни).
- Клиент получает cookie с идентификатором сессии: атрибуты **HttpOnly**, **Secure** (в продакшене), **SameSite** (Lax или Strict по среде).
- **BFF** по cookie определяет пользователя (валидация сессии через IAM или прямое чтение Redis по согласованному контракту).

JWT в `localStorage` для веб-клиента **не используем** на MVP: ниже риск кражи сессии при XSS; выход из системы инвалидирует сессию на сервере для всех вкладок.

### 5. Межсервисная идентичность

- С публичного интернета принимаются только запросы к Gateway/BFF.
- BFF при вызовах к микросервисам передаёт **доверенный** признак пользователя: заголовок `X-User-Id` (UUID) и/или внутренний сервисный JWT **только внутри** Docker-сети / приватной сети.
- Микросервисы **не доверяют** `user_id` из тела запроса как владельцу ресурса без сопоставления с этим доверенным контекстом.
- Дополнительно: общий секрет `X-Internal-Auth` для вызовов из BFF (не принимать с публичных интерфейсов). **mTLS** между сервисами — опциональное усиление после MVP.

## Последствия

- Положительные: чёткие границы владения данными, единая дверь для браузера, прослеживаемость запросов по correlation id.
- Отрицательные: операционная сложность (несколько процессов, несколько БД), необходимость дисциплины в OpenAPI и интеграционных тестах.
