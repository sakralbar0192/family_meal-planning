# ADR 0002: Агрегат плана, рецепт в слоте и снимок в списке покупок

## Статус

Принято

## Контекст

Нужно определить:

- как хранить назначение рецепта в планировщике относительно изменяемого каталога рецептов;
- как обеспечить предсказуемый список покупок после формирования ([business-doc.md](../business-doc.md) §3.3);
- границу транзакции при операциях DnD ([architecture-decision-context.md](../architecture-decision-context.md) §6 п.8).

## Решение

### 1. Планировщик (`meal-planning`): ссылка на рецепт в слоте

- В слоте хранится **идентификатор рецепта** (`recipe_id`), ссылающийся на сущность в `recipe-catalog`, а не полная копия рецепта.
- Отображаемое имя и метаданные для UI могут подгружаться из каталога (read-модель / BFF-агрегация).
- Если рецепт **удалён** из каталога, слот остаётся с «битой» ссылкой: UI показывает состояние «рецепт недоступен» и предлагает убрать из слота или заменить (согласовать с [design-plan.md](../design-plan.md)).

### 2. Список покупок (`shopping-list`): снимок состава при формировании

- По команде «Сформировать список» сервис строит агрегат из плана за период и **фиксирует снимок** ингредиентов (название, количество или «по вкусу», единица, категория продукта) **на момент формирования**.
- Последующее **изменение рецепта** в каталоге **не изменяет** уже сохранённый список задним числом. Новый расчёт — только по новой команде пользователя (повторное формирование).
- Хранение снимка: отдельные строки таблицы `shopping_list_item` (или эквивалент) с денормализованными полями; альтернатива — JSONB по строке, если команда согласует единый формат в OpenAPI.

### 3. Повторное формирование за тот же период

- **Политика MVP:** идемпотентная команда с тем же периодом **заменяет** текущий рабочий список для этого периода (или создаёт новую версию — выбрать одно и зафиксировать в OpenAPI). Рекомендация MVP: **replace** одного активного списка на период для упрощения UI.

### 4. Граница агрегата и транзакции при DnD

- **Агрегат:** недельный контур (`WeekPlan`) или дневной (`DayPlan`) — команда выбирает минимальную границу, внутри которой инварианты слотов согласованы.
- **Транзакция:** одна операция API «перенести/добавить/удалить назначение» изменяет состояние в рамках **одного агрегата** (например один день или одна неделя), чтобы избежать частично применённого DnD при ошибке.
- API назначений — **идемпотентные** там, где уместно (например `PUT` слота с ожидаемой версией или ETag), чтобы повтор запроса с клиента не дублировал блюда.

### 5. Связь план ↔ список (MVP)

- Синхронная команда в `shopping-list`: чтение назначений у `meal-planning` и при необходимости актуального состава рецептов у `recipe-catalog` **в момент формирования** (до записи снимка).
- Асинхронные события и очереди — вне scope MVP.

## Последствия

- Пользователь доверяет списку в магазине: он не «прыгает» при правке рецепта после расчёта.
- План остаётся лёгким по хранению; требуется UX для удалённых рецептов.
