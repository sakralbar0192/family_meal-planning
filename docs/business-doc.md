### 1. Цель и обоснование
Создать веб-приложение, которое позволяет пользователям:
- управлять рецептами (создавать, редактировать, импортировать других сайтов (в рамках MVP - с двух конкретных сайтов));
- планировать приёмы пищи на календарной основе;
- автоматически формировать список покупок на основе выбранного периода.
- в будущем планируется добавление напоминаний и оповещений для мобильной версии

Основная ценность — упрощение организации домашнего питания за счёт централизации рецептов, удобного планировщика и автоматического списка покупок.

### 2. Целевая аудитория
- Люди, которые готовят дома и хотят систематизировать рецепты.
- Пользователи, привыкшие находить рецепты на популярных кулинарных сайтах.
- Пользователи, ценящие время: им нужен быстрый импорт, планирование меню и автоматический список покупок без ручного ввода.

### 3. Основные функциональные требования

#### 3.1. Модуль рецептов
- **Создание рецепта вручную:** пользователь может заполнить все поля (название, ингредиенты с количеством и единицами измерения, шаги, время готовки, пищевая ценность, категория приемов пищи).
- **Редактирование рецепта:** возможность изменять любые поля созданного или импортированного рецепта.
- **Добавление ингридиентов:** для каждого ингредиента должно быть указано количество (или оставлено "по вкусу"), а так же указано или выбранны из существующих единицы измерения (в отдельных полях), а так же категория продуктов (молочные, мясо, бакалея, овощи и тд).
- **Импорт рецепта по URL:** пользователь вставляет ссылку на рецепт с сайта. Система парсит страницу и заполняет поля рецепта (название, ингредиенты, шаги, время, фото при наличии). Пищевая ценность импортируется, если доступна на странице, в противном случае остаётся пустой для ручного заполнения.
- **Пищевая ценность:** может быть указана на приём пищи (белки, жиры, углеводы, калории). Для импортированных рецептов данные сохраняются как есть, без пересчёта.
- **Хранение рецептов:** все рецепты сохраняются в личной библиотеке пользователя.

#### 3.2. Планировщик питания
- **Календарь:** отображение понедельной сетки с возможностью выбора конкретного дня.
- **Слоты приёмов пищи:** для каждого дня предусмотрены слоты (в MVP - фиксированные: завтрак, второй завтрак, обед, полдник, ужин, поздний ужин ).
- **Добавление рецептов:** пользователь перетаскивает рецепт из бокового списка (или нажимает «добавить») в нужный слот выбранного дня.
- **Добавление из экрана рецепта:** по **«В план»** открывается **месячный календарь**; после **выбора дня** открывается планировщик, в **поле поиска** боковой панели рецептов **подставляется название текущего рецепта**, чтобы пользователь мог сразу найти карточку и **перенести её в слот** (или воспользоваться альтернативой без drag-and-drop на тач-устройствах).
- **Фильтрация рецептов:** рецепты из бокового списка при добавление рецептов, а так же в быстром поиске на странице библиотеки рецептов можно фильтровать по названию, времени приготовления, типу блюда и т.п. (базовая фильтрация для удобства).
- **Редактирование плана:** можно удалить рецепт из слота, заменить на другой, переместить в другой день.

#### 3.3. Списки покупок
- **Формирование списка:** на основе выбранного периода (например, неделя, указанные дни) система агрегирует все ингредиенты из рецептов, назначенных на эти дни.
- **Объединение одинаковых ингредиентов:** если один и тот же ингредиент встречается в нескольких рецептах, их количества суммируются (с учётом единиц измерения; если единицы разные - система приводит их к одной (если логика заложена в коде) либо указывает в виде перечисления).
- **Просмотр и редактирование списка:** пользователь может отметить купленные продукты, удалить лишнее, добавить свои позиции.
- **Экспорт:** возможность скопировать список в буфер обмена (в виде текста или простого списка).

### 4. Нефункциональные требования
- **Платформа:** адаптивная веб-версия (работает на десктопах, планшетах, мобильных устройствах через браузер и как мобильное приложение).
- **Хранение данных:** данные пользователя хранятся на сервере (требуется регистрация/аутентификация). Для MVP — базовая аутентификация (email/пароль).
- **Производительность:** импорт рецепта должен занимать не более 3–5 секунд; планировщик должен работать без заметных задержек при перетаскивании.
- **Надёжность парсинга:** импорт с сайтов должен работать стабильно.
- **UI/UX:** интерфейс интуитивно понятен, drag-and-drop поддерживается на всех устройствах (для десктопа — мышью, для тач-устройств — сенсорное перетаскивание).

### 5. Ограничения (Scope)
- На первом этапе поддерживается импорт только с двух конкретных сайтов (их названия/адреса фиксированы).
- Планировщик использует предопределённый набор приёмов пищи. Настройка слотов пользователем — в будущих версиях.
- Список покупок не учитывает остатки продуктов на складе (просто агрегация).
- Нет интеграции с внешними сервисами доставки продуктов.
- Мобильное приложение через адаптацию веб версии.

### 6. Пользовательские сценарии (Use Cases)

#### Сценарий 1: Импорт рецепта с сайта
1. Пользователь авторизуется в приложении.
2. Переходит в раздел «Рецепты» и нажимает кнопку «Импорт по URL».
3. Вставляет ссылку на рецепт и нажимает «Импортировать».
4. Система проверяет, что URL принадлежит разрешённому сайту, загружает страницу и извлекает данные:
   - название;
   - ингредиенты (название, количество, единица измерения);
   - шаги приготовления;
   - общее время;
   - пищевую ценность (если есть).
5. Открывается форма редактирования рецепта с заполненными полями. Пользователь может отредактировать данные (например, исправить единицы измерения, добавить заметки) и нажать «Сохранить рецепт».
6. Рецепт сохраняется в библиотеке.

*Альтернативный поток:* если URL не поддерживается, система выводит сообщение об ошибке и предлагает ввести рецепт вручную.

#### Сценарий 2: Планирование питания на неделю
1. Пользователь открывает планировщик питания.
2. Календарь отображает текущий неделю. Пользователь выбирает день (например, понедельник).
3. В правой панели видит список своих рецептов. С помощью фильтра (поиск по названию, приему пищи, времени готовки) находит нужный рецепт.
4. Перетаскивает рецепт мышью (или нажимает «в план» и выбирает слот) в слот «ужин» для понедельника.
5. Рецепт отображается в слоте. При необходимости пользователь может перетащить другой рецепт дополнительно. Удаление рецептов — нажатием на значок корзины.
6. Повторяет для других дней недели. План автоматически сохраняется.

#### Сценарий 3: Составление списка покупок на выбранный период
1. Пользователь находится в планировщике и в форме выбора дат для списка покупок выбирает диапазон дат.
2. Нажимает кнопку «Сформировать список покупок».
3. Система собирает все ингредиенты из рецептов, назначенных на выбранные дни, группирует и суммирует одинаковые позиции.
4. Открывается страница списка покупок, где ингредиенты сгруппированы по категориям (овощи, молочные и т.п. — если категории указаны в рецептах, иначе просто алфавитный список).
5. Пользователь может отметить галочкой купленные позиции, добавить свои строки, удалить ненужные.
6. Нажимает «Экспортировать» — система копирует список в буфер обмена в виде текста.

*Альтернативный поток:* если на выбранный период нет ни одного рецепта, система уведомляет, что список пуст, и предлагает добавить рецепты в планировщик.

### 7. Критерии успеха (на MVP)
- Пользователь может успешно импортировать рецепты с двух целевых сайтов (не менее 90% успешных импортов для корректно структурированных страниц).
- Планировщик позволяет составить меню на неделю за 5–10 минут.
- Список покупок для недельного меню формируется автоматически и корректно суммирует ингредиенты.
- Интерфейс адаптивен и удобен на мобильных устройствах (базовая навигация, работа drag-and-drop).

### 8. Дальнейшее развитие (за рамками MVP)
- Поддержка дополнительных сайтов.
- Настройка приёмов пищи и слотов пользователем.
- Мобильные приложения (iOS/Android).
- Умное предложение рецептов на основе истории.
- Интеграция с доставкой продуктов.
- Возможность делиться планами и рецептами.
