# Контекст для архитектурных решений

## 1. Назначение и статус

Этот документ — **единая входная точка** при проектировании solution architecture и при оформлении **ADR** (Architecture Decision Records). Он **не заменяет** исходные спецификации и **не фиксирует** выбор стека, БД или схемы деплоя.

**Важно:** ограниченные контексты и события, на которые здесь даются отсылки, описаны в [domain-contexts-event-storming.md](./domain-contexts-event-storming.md) как **рабочие гипотезы** до воркшопа по Event Storming. После уточнения границ на сессии следует обновить доменный документ и при необходимости этот файл.

**Связанные артефакты:** [README.md](./README.md) (оглавление `docs/`), [business-doc.md](./business-doc.md), [domain-contexts-event-storming.md](./domain-contexts-event-storming.md), [design-plan.md](./design-plan.md), [technical-spec-figma.md](./technical-spec-figma.md).

## 2. Цели и границы MVP

**Продукт:** веб-приложение для личной библиотеки рецептов, планирования приёмов пищи по календарю и автоматического списка покупок на выбранный период. Ценность — централизация рецептов, быстрый импорт с ограниченного набора сайтов, план на неделю без ручной агрегации покупок.

**В MVP (кратко):**

- Рецепты: CRUD, ингредиенты с единицами и **категорией продукта**, импорт по URL **только с двух фиксированных сайтов**, пищевая ценность как в источнике / вручную.
- Планировщик: недельная сетка, **шесть фиксированных слотов** приёма пищи, назначение рецептов (DnD и альтернативы для тача), сценарий «В план» с экрана рецепта.
- Список покупок: период → агрегация ингредиентов по плану, суммирование одинаковых позиций (с учётом единиц или явное перечисление при несовместимости), ручные правки, экспорт в буфер обмена.
- Аутентификация: **email/пароль**, данные на **сервере**.

**Вне MVP (не проектировать без отдельного решения):** произвольные сайты импорта, настраиваемые слоты, склад/остатки, доставка, нативные приложения (в перспективе — PWA/оболочка веба), напоминания, шаринг планов.

**Критерии успеха MVP** (дословно по смыслу из бизнес-документа): стабильный импорт с двух сайтов (ориентир ≥90% для корректных страниц), план на неделю за 5–10 минут, корректная агрегация списка, адаптивный UI с рабочим DnD. Подробнее: разд. 7 [business-doc.md](./business-doc.md).

## 3. Качественные атрибуты (из NFR)

| Атрибут | Ожидание (источник) | Заметки для архитектуры |
|--------|----------------------|-------------------------|
| **Производительность** | Импорт: 3–5 с; планировщик без заметных задержек при DnD ([business-doc.md](./business-doc.md) §4) | Где выполняется парсинг (синхронно / асинхронно), лимиты таймаутов HTTP к внешним сайтам, стратегия оптимистичного UI. |
| **Надёжность интеграции** | Импорт с целевых сайтов стабилен ([business-doc.md](./business-doc.md) §4) | Версионирование парсеров, деградация при изменении вёрстки, наблюдаемость ошибок. |
| **Безопасность и изоляция данных** | Данные пользователя на сервере; регистрация/вход ([business-doc.md](./business-doc.md) §4) | Модель tenant (пользователь), хранение учётных данных, сессии vs токены — см. открытые вопросы. |
| **Доступность клиента** | Понятный UI; DnD на десктопе и таче ([business-doc.md](./business-doc.md) §4) | Согласовать с токенами и фокусом: `color/focus-ring`, `touch-target` в [technical-spec-figma.md](./technical-spec-figma.md). |
| **Масштабируемость** | Явно не задана для MVP | Задать допущения (число пользователей, рост данных) в первом ADR по платформе. |

Контексты и типы связей — гипотезы; детали и тактика — в [domain-contexts-event-storming.md](./domain-contexts-event-storming.md) §1–3. Единый язык — §2 того же документа.

| Bounded context | Ответственность | Ключевые связи с другими контекстами | Визуализация |
|-----------------|-----------------|--------------------------------------|--------------|
| **Identity & Access** | Регистрация, вход, сессия, минимальный профиль | Customer–Supplier → **Recipe Catalog** (данные привязаны к пользователю) | [domain-contexts-event-storming.md](./domain-contexts-event-storming.md) §1.2, Figma DDD |
| **Recipe Catalog** | Личная библиотека рецептов, CRUD, метаданные | Принимает черновики от **Recipe Import** (Conformist); Partnership / C–S ↔ **Meal Planning** | то же |
| **Recipe Import** | Whitelist URL, парсинг, маппинг в черновик рецепта | Supplier → **Recipe Catalog**; внешние HTTP к двум сайтам | то же; *альтернатива:* подпроцесс внутри Catalog |
| **Meal Planning** | Недельный план, слоты, назначения рецептов | Зависит от каталога; Customer–Supplier → **Shopping List** (период и план как вход) | то же |
| **Shopping List** | Агрегация на период, группировка, правки, экспорт | Вход из плана и периода; внешний **буфер обмена** при экспорте | то же |

**Доска в Figma (страница DDD):** секция «DDD — Event Storming и карты контекстов», node id `3372:1312`; фрейм **Board root** `3372:1314` — см. конец [domain-contexts-event-storming.md](./domain-contexts-event-storming.md).

**Открытый вопрос по стратегии:** нужен ли **ACL** между Import и Catalog при сильном расхождении моделей источника и рецепта ([domain-contexts-event-storming.md](./domain-contexts-event-storming.md) §1.2).

### 5.2. UX-инварианты, влияющие на бэкенд и контракты

Из [design-plan.md](./design-plan.md) и [technical-spec-figma.md](./technical-spec-figma.md) (навигация):

- Выбор **периода для списка покупок** — только в контексте **планировщика** (не библиотека / не карточка рецепта); **единый источник правды для дат** — планировщик. Со списка **«Изменить период»:** в [design-plan.md](./design-plan.md) — рекомендация вернуться в планировщик; в текущем Figma-прототипе задокументирован **оверлей календаря** — см. [technical-spec-figma.md](./technical-spec-figma.md).
- **«В план»** с экрана рецепта: выбор дня в календаре → открытие планировщика с **предзаполненным поиском** по названию рецепта (состояние/параметры навигации или query).
- Импорт: явная валидация поддерживаемого URL и сообщение об ошибке — согласовано с модальными состояниями в ТЗ Figma.

## 6. Открытые архитектурные вопросы (кандидаты под ADR)

Варианты **не выбираются** в этом документе; каждый пункт — кандидат на отдельный ADR или подраздел solution design.

1. **Recipe Import:** отдельный deployable / модуль монолита / подпроцесс Catalog; необходимость **ACL** и границы транзакции «импорт → сохранение рецепта».
2. **Meal Planning ↔ Shopping List:** синхронный пересчёт списка по команде vs доменные события и подписка; **полная пересборка** списка при изменении плана/периода vs инкрементальное обновление ([domain-contexts-event-storming.md](./domain-contexts-event-storming.md) §3.5).
3. **Парсинг:** выполнение в запросе API vs фоновая задача/очередь vs отдельный worker; лимиты и отмена долгих операций при SLA 3–5 с.
4. **Агрегация ингредиентов:** стратегия **нормализации единиц** (или явный отказ с перечислением строк) — влияет на доменную модель и хранение.
5. **План и рецепты:** снимок состава рецепта в плане vs ссылка на актуальную версию; поведение при **изменении/удалении** рецепта после назначения в слот.
6. **Аутентификация:** cookie-сессии vs JWT; требования к выходу, смене пароля, будущему сбросу по почте ([domain-contexts-event-storming.md](./domain-contexts-event-storming.md) §3.1).
7. **Мультитенантность:** пользователь как единственный изолированный арендатор на MVP; границы данных в API и БД.
8. **Граница агрегатов:** WeekPlan / DayPlan / Slot и граница транзакции при DnD ([domain-contexts-event-storming.md](./domain-contexts-event-storming.md) §3.4) — влияет на API и конкурентные обновления.
