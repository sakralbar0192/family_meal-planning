# Техническое задание: интерактивный макет веб-приложения планирования питания

**Оглавление папки `docs/`:** [README.md](./README.md).

## Принятые решения

- **Язык интерфейса в макетах и ТЗ:** русский.
- **Файл Figma:** по задумке — отдельный новый файл; фактически первичная сборка выполнена в подключённом через MCP файле (см. раздел «Ограничения среды»). Для продакшн-гайда рекомендуется **скопировать страницы** в новый файл команды или обновить тариф Figma.
- **Иконки:** рекомендуется набор **[Lucide](https://lucide.dev/)** (открытая лицензия, единый стиль с минималистичным UI).
- **Карточка заметки (`DS / NoteCard`):** используется в **экране просмотра рецепта** (краткая заметка / подсказка) и может дублироваться в **планировщике** для заметок к дню; отдельно от карточки рецепта.
- **Доменные границы и материал для Event Storming:** [domain-contexts-event-storming.md](./domain-contexts-event-storming.md) (стратегическая карта контекстов, единый язык, тактические заготовки). Бизнес-правда по функциям — в [business-doc.md](./business-doc.md).

## Ограничения среды (зафиксировать для команды)

1. **Тариф Figma Starter:** доступно **не более 3 страниц** в файле. Структура разнесена так:
   - `00 — IA` — информационная архитектура (схема блоков).
   - `01 — Токены и UI Kit` — секции **A. Токены**, **B. UI Kit**.
   - `Page 1` — **экраны-фреймы верхнего уровня** для прототипа: цели переходов **`NAVIGATE`**, **`OVERLAY`**, **`SWAP`** в Figma должны быть **прямыми детьми страницы** (не вложены в **SECTION** и т.п.). Иначе Present и MCP-прототип отклоняют оверлей или ведут себя как полноэкранная смена кадра.
2. **Переменные:** в текущем файле коллекция поддерживает **один режим** (нельзя добавить второй `Dark` в ту же коллекцию). Светлая тема задана переменными коллекции **`Notes / Tokens`**. Значения **тёмной темы** приведены в таблице ниже — для реализации в коде и для ручного перекрашивания тёмных макетов. В документе через MCP зафиксированы **две** записи с именем `Notes / Tokens` (одна без переменных) и пустая **`Питание — токены`** — при уборке файла оставить одну рабочую коллекцию; в рабочей коллекции **38** переменных (см. список ниже).
3. **Шрифты:** используется **Inter** (после настройки локальных шрифтов в Figma Desktop). В макетах и мастер-компонентах добавлен **русский текст** (кнопки, поля, карточки, планировщик, список покупок, модальная оболочка). При смене машины без Inter повторите установку шрифта или замените семейство через меню Figma.

## Этап 1. Анализ требований и ИА

### Разделы приложения

1. Вход / регистрация (упрощённо).
2. Главная / дашборд (опционально объединить с планировщиком и недавними рецептами).
3. Библиотека рецептов.
4. Просмотр рецепта.
5. Создание / редактирование рецепта.
6. Импорт по URL.
7. Планировщик питания (неделя, слоты приёмов пищи).
8. Список покупок.
9. Профиль / настройки (минимум).

### Ключевые переходы

- Вход → дашборд / библиотека.
- Библиотека → карточка рецепта (клик по телу) → просмотр; импорт; на **desktop** и **tablet** кнопка **«Планировщик»** → **оверлей с календарём** (`MOD / Календарь` / **`MOD / Календарь — Tablet 768`**) → **клик по числу дня** → планировщик; на **mobile** тот же макет календаря оформлен как полноэкранный **`SCR / Календарь — Mobile 390`** (**`NAVIGATE`** из библиотеки — смена экрана, без стека оверлея); чип **«В план»** на карточке → по продуктовой спецификации — модалка **«Добавить в план»** (см. **«Статус синхронизации»** — фреймы модалки в текущем файле могут отсутствовать, прототипные связи перепроверять в Figma).
- **Планировщик** → панель **«Список покупок на период»** → **«Сформировать список покупок»** → список покупок (основной сценарий MVP).
- **Список покупок** → **«Изменить период»** / «К планировщику» → планировщик (единый источник правды для дат).
- Рецепт (просмотр) → **«В план»** → **оверлей месячного календаря** (`MOD / Календарь — из рецепта`) → **клик по дню** → **`SCR / Планировщик — из рецепта`**: в поле поиска панели рецептов **уже введено название этого блюда** (в продукте — автоподстановка; в макете — демо-текст), далее перенос карточки в слот.

### Состояния на экранах

Для каждого экрана в финале макета: **пусто**, **загрузка**, **ошибка**, **успех** (хотя бы один вариант на экран в backlog после MVP-прототипа).

**Артефакт в Figma:** страница `00 — IA`, фрейм с номерованными блоками экранов; добавлен текстовый блок с пояснением потоков **панель периода → список покупок**, **«В план» с карточки → модалка** и **«В план» с экрана рецепта → календарь → планировщик с поиском** (`3355:590` внутри `IA — поток экранов…`, аудит 2026-03-22 — при расхождении сверять с [design-plan.md](./design-plan.md) п. 3).

### DS / Input / Field — выравнивание с кнопками

**Проблема:** у мастера два блока по вертикали — **подпись** (`Label …`) и **поле** (`Field`). Высота инстанса больше, чем у **`DS / Button / *`** (**`button/min-height`** = **44**). В горизонтальном `auto layout` с **`counterAxisAlignItems: CENTER`** центрируется весь инстанс, а не строка ввода — ряд с кнопками выглядит «сломанным».

**Реализация в файле (мастер `DS / Input / Field`, `3339:105`):**

- Добавлено свойство компонента **`Show label`** (тип **BOOLEAN**, по умолчанию **true**).
- Фрейм **`Label (текст в макете)`** (`3339:106`) привязан к этому свойству: **`visible`** ↔ **`Show label`** (через `componentPropertyReferences` на слое подписи).
- При **`Show label = false`** подпись не участвует в вёрстке, высота инстанса совпадает с **полем** (**`minHeight` 44**), выравнивание с кнопками в одной строке корректное.
- Инстансы в **тулбарах** (все найденные в прототипе с родителем `Toolbar`) переключены на **`Show label = false`**; смысл бывшей подписи переносится в **плейсхолдер** поля там, где нужно.
- Инстансы в **боковой панели планировщика**, **формах** и **модалках** оставьте с **`Show label = true`**, если нужна внешняя подпись.

*После копирования файла* имя ключа свойства в API может стать вроде `Show label#<id>` — в UI Figma отображается как **Show label**.

- **Календарь на touch:** визуально тот же паттерн (scrim **`color/overlay`** + каркас **`DS / Modal / Shell`**). **Tablet 768:** в прототипе **`OVERLAY`** поверх библиотеки / планировщика (фреймы **`MOD / Календарь … — Tablet 768`** — **прямые дети `Page 1`**, не внутри секции, иначе Figma не даст **`OVERLAY`**). **Mobile 390:** календарь из библиотеки может оставаться **`NAVIGATE`** на **`SCR / Календарь — Mobile 390`** (полноэкранный кадр); **«В план»** с рецепта — **`OVERLAY`** на **`MOD / Календарь — из рецепта — Mobile 390`** (или эквивалент на корне страницы).
- **Прототип:** связи **только внутри одной ширины** (768→768, 390→390), чтобы Present не перескакивал на desktop.

## Примечание по MCP (vibma)

1. `connection.create` / `connection.get` — связь с плагином в Figma.
2. `document` — страницы (лимит Starter).
3. `variable_collections` / `variables` — токены.
4. `components` / `instances` — UI Kit и экраны.
5. `frames` — секции, фреймы, **create** (вызов `method:"create", type:"auto_layout", items:[...]`), **clone/update/delete/reparent/export**.
6. `prototyping` — реакции на **экземплярах, фреймах, тексте**; цель **`NAVIGATE`** — фрейм **прямой потомок страницы** (не вложен в `SECTION` — иначе плагин vibma отклоняет связь); **`OVERLAY`** — топ-уровневый фрейм оверлея. Часть оверлеев (импорт, календарь из рецепта **tablet**) лежит **прямым ребёнком `Page 1`**; экран **`SCR / Рецепт`** (`3339:192`) при необходимости дублируется на корне страницы для **`NAVIGATE`** из редактора (см. актуальную иерархию `frames.get(Page 1)`).
7. `lint` — проверка токенов и композиции.
